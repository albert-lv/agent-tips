## 技巧一：Claude Code 的 TDD 三步循环

你有没有发现，跟 Agent 说"帮我写个函数"，它写得挺快，但你心里总有点虚——这玩意儿真的对吗？

**解法：让 Agent 先写测试，再写实现，再自己跑一遍。** Claude Code 对这个 workflow 支持得特别好：

```bash
# 第一步：让它写测试（描述清楚你要什么）
claude "给 `calculateDiscount` 写几个测试，覆盖正常折扣、零折扣、和负数输入的边界情况"

# 第二步：让它写实现（只看测试，不猜需求）
claude "根据刚才的测试，实现 `calculateDiscount` 函数，确保所有测试通过"

# 第三步：让它跑测试并修复
claude "跑一下测试，如果有失败的，修到全绿为止"
```

Claude Code 会：
1. 读你项目的测试框架（Jest、pytest、Go test 等）
2. 写测试用例
3. 写实现代码
4. 在终端里跑 `npm test` / `pytest` / `go test`
5. 看到报错后自动分析、修改、再跑——循环到全绿

**为什么这招香：**
- 你不需要盯着它写，它自己会跑会修
- 测试就是需求文档，Agent 不会"过度发挥"
- 你最后拿到的是**带测试的、跑过的**代码，不是草稿

> 💡 进阶：直接说 "帮我实现 TDD 风格的折扣计算模块，先写测试再写代码"，Claude 一步搞定整个循环。

## 技巧二：把终端报错直接甩给 Agent（Cursor / Kimi Code）

跑测试挂了，报错信息十几行，你懒得看？**直接复制粘贴给 Agent。**

**Cursor 的做法：**
在终端里跑测试，报错后选中错误信息，按 `Ctrl+Shift+L`（或右键 "Add to Chat"），Agent 会自动读报错、定位文件、给修复方案。

```
你：
[粘贴报错]
"这个测试挂了，帮我修一下"

Cursor Agent：
- 分析了错误：`assertion failed: expected 42, got 0`
- 定位到 `src/calc.js:15`：`return a - b` 应该是 `return a + b`
- 已修复，跑了测试确认全绿
```

**Kimi Code 的做法：**
Kimi Code 的终端集成支持直接把命令输出塞进对话。跑完测试后，对 Kimi 说：
> "刚才终端跑 `pytest` 的输出，分析一下为什么挂了，然后修"

Kimi 会读取最近的终端输出，自动找 bug。

**为什么这招香：**
- 你不需要解释 "这个函数应该返回什么"——测试和报错已经说清楚了
- Agent 会自己读堆栈、找文件、改代码
- 相当于雇了一个会自己看报错、自己调试的实习生

## 🤝 组合技：TDD + 自动修复 = 无痛开发

一个日常 workflow：

1. **Claude Code** 写功能和测试（TDD 循环）
2. **Cursor** 在编辑器里继续跑测试，挂了就把报错丢给 Agent 修
3. 你负责**看结果**和**拍板合并**，中间写代码和调试的 dirty work 全交给 Agent

相当于：**你当 PM 提需求，Agent 当全栈工程师写代码+测试+调试。**
