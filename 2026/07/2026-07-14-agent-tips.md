# Agent 改崩了？你的「紧急制动」指南 🚨

> 2026-07-14 | 每日 Agent Tips

你有没有经历过这种心跳骤停的瞬间：

让 Claude Code 改个小功能，它热情满满地改了 30 个文件，最后来一句「好了，运行一下看看」——

你跑了一下，**整个项目炸了。** 💀

不是它故意搞你，是 Agent 有时候太自信了。它会"顺便"重构一下你觉得没问题的代码，"顺手"改一下配置文件，然后一脸无辜地看着你。

今天教你三招**紧急制动**，让 Agent 在失控之前就能被拉住。

---

## 技巧一：改前先「拍快照」

Agent 不会自己先 git commit 再动手——但你可以要求它这么做。

### Claude Code 的做法

```bash
claude "在开始任何修改之前，先 git stash 保存当前状态，并告诉我 stash 的编号"
```

如果改崩了，直接 `git stash pop` 或者 `git checkout .` 就能秒回干净状态。

### 更狠的：让 Agent 自己建 checkpoint

```bash
claude "每完成一个独立的修改步骤，就 git commit 一次，message 写清楚改了什么。如果后面改崩了，我们要能一步步回退"
```

这样 Agent 会自己维护一条"时间线"，而不是把所有改动堆在一个巨大的 diff 里。

| 做法 | 改崩了怎么办 |
|------|------------|
| 让 Agent 自由发挥 | `git diff` 看几千行，人工回退到哭 |
| 改前先 stash | `git stash pop` 一键回退 |
| 让 Agent 逐步 commit | `git reset HEAD~3` 精准撤销 |

## 技巧二：给 Agent 画个「禁区地图」

Agent 有时候会"顺手"改一些它觉得"顺便优化一下"的文件——然后你就debug到凌晨了。

### 在 prompt 里直接划红线

```
修改范围限制：
- 只改 src/components/ 下的文件
- 不要动 package.json、tsconfig.json 或任何配置文件
- 不要重构与本次需求无关的代码
- 每改一个文件，告诉我改了什么，不要一次性改完再汇报
```

### 进阶：项目级「禁区清单」

在你的项目根目录放一个 `.agent-guardrails` 文件（Claude Code 和 Cursor 都会读）：

```
# Agent 禁区
DO NOT MODIFY:
- package-lock.json / yarn.lock / pnpm-lock.yaml
- .env 或任何包含密钥的文件
- 第三方库的源码（node_modules/ 等）
- CI/CD 配置文件（除非明确要求）

DO NOT REFACTOR:
- 与当前需求无关的代码
- 正在 review 中的 PR 相关文件
- 任何标有 @deprecated 的接口（除非要删除它们）
```

这样每次 Agent 启动时，它都会先读这个文件，知道哪些东西是碰不得的。

## 技巧三：学会「中途叫停」

有时候 Agent 已经开始跑了，你发现它正在往坑里冲——这时候怎么拦？

### 大部分 Agent 工具的中断方式

| 工具 | 怎么叫停 |
|------|---------|
| Claude Code | `Ctrl+C` 中断当前操作，然后说 "stop，刚才的方向不对" |
| Kimi Code | 点击停止按钮，或用 "取消当前操作" |
| Cursor Agent | `Escape` 键或点击取消 |
| Windsurf | 点击停止生成 |

### 叫停之后怎么说？

不要只说"不对"，要**给方向**：

❌ 不好的说法：
> "错了，重来"

✅ 好的说法：
> "停，刚才改到 `auth.ts` 那步就跑偏了。我需要的是在前端加校验，不是改后端逻辑。从 stash 恢复，重新来，只改 `src/pages/login.tsx`"

**关键**：叫停 + 给原因 + 给正确方向 + 给范围限制。四件套一出，Agent 基本不会再跑偏。

---

## 一句话总结

Agent 不是故意的，它就是太热心了。**你的刹车踩得越及时，它帮你开得越远。**

下次让 Agent 动手之前，先丢给它一句：

> "先 stash 一下，改完一步汇报一步，别碰配置文件。"

这三句话，能救你一晚上。
