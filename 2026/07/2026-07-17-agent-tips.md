# Agent 的"多方案对比"——别只给一个答案

> 2026-07-17 | 每日 Agent Tips

你有没有发现，每次问 agent "这个功能怎么实现"，它总是给你一个"标准答案"——然后你用了，过两天发现"其实还有更好的方式"...

不是它藏着掖着，是你没让它**多给几个选项**。

---

## 技巧一：一键开启"多方案模式"

大部分 agent 默认是"答题模式"：你问什么，它给一个最优解。但编程里很少有唯一最优解，更多时候是"权衡"。

试试在你的需求前面加一句：

> "请给出 2-3 种实现方案，分别说明优缺点和适用场景。最后推荐一种，并说明理由。"

Agent 的输出会完全不同：

| 普通模式 | 多方案模式 |
|---------|-----------|
| "用 Redis 缓存" | "方案 A：Redis（快，但需要额外依赖）<br>方案 B：内存 Map（简单，但重启丢失）<br>方案 C：SQLite（持久化，但比 Redis 慢）" |
| 直接给代码 | 先给思路，让你选 |

**为什么这招好用？**

- 你能看到"简单版"和"完整版"的区别，按需选择
- 避免过度工程——有时候 agent 给的方案太"重"了
- 学习视角——对比过程本身就是很好的学习材料

### 实战 prompt 模板（收藏）

```
我要实现 [功能描述]。
请给出 2-3 种实现方案，要求：
1. 每种方案给出核心代码片段（不超过 20 行）
2. 说明优缺点和适用场景
3. 指出每种方案可能踩的坑
4. 最后推荐一种，并说明理由

注意不要给我"过度工程"的方案，优先考虑可维护性。
```

## 技巧二：让 Agent 当"方案分析师"

如果你已经有几个方案在脑子里，可以让 agent 帮你做对比分析：

> "我在纠结两种方案：
> 1. 用 WebSocket 做实时推送
> 2. 用 Server-Sent Events
> 
> 从实现复杂度、浏览器兼容性、服务器资源消耗三个维度分析一下，给我建议。"

Agent 会输出一个清晰的对比表，甚至比你自己查资料还快。

**进阶玩法：让 Agent 预测"未来的你"会怎么选**

> "假设三个月后的项目规模翻了 10 倍，哪个方案更经得起考验？"

Agent 会切换到"长期视角"，提醒你那些现在看起来不重要但以后很要命的因素。

---

## 适用场景

| 场景 | 为什么用多方案模式 |
|------|-----------------|
| 技术选型 | 避免一叶障目，看到权衡全貌 |
| 重构决策 | 是"小步调整"还是"大改"，agent 能给出梯度方案 |
| 性能优化 | 从"低成本改善"到"终极方案"列出来，按预算选 |
| 学习新技术 | 对比过程本身就是最好的教程 |

---

## 一句话总结

> **Agent 不是不会给多个方案，是默认在等你的许可。** 你问得开放，它回得丰富；你问得封闭，它回得单一。

下次觉得 agent 给的方案"有点太标准了"，试试加一句"给几个选项我看看"——它真的能。
