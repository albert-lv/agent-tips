# 🎭 Agent 的「多开」技巧——为什么同时开三个会话反而更快

> 日期：2026-06-22（周一）  
> 致：向上 & 小惠 —— 今天教你们一个「作弊级」效率技巧

---

## 先讲个现象

你有没有跟 Claude Code 或 Kimi Code 聊了一下午，最后发现：

- 前半段在聊架构设计
- 中间突然变成 debug 某个报错
- 后半段又跑回去改早上那块的命名规范
- 结果 Agent 一脸懵：「等等，咱们现在到底在干嘛？」

这就像你跟一个朋友同时聊三个话题——每个人都会宕机。

**解法简单到离谱：多开几个会话。**

不是让你买会员，而是**同一个项目，开 2-3 个 Agent 窗口，各干各的。**

---

## 技巧一：「三窗口工作流」

这是我的常用配置，复制即用：

| 窗口 | 职责 | 示例对话 |
|------|------|---------|
| **窗口 A** —  architect（建筑师） | 只聊设计，不碰代码 | "这个模块的接口怎么设计比较合理？" "画个调用流程图" |
| **窗口 B** —  builder（施工队） | 只实现，不问为什么 | "按刚才的设计，把 `userService.ts` 写出来" |
| **窗口 C** —  inspector（质检员） | 只 review 和测试 | "看看这段代码有没有边界情况没处理" "写个测试覆盖这个函数" |

**为什么分开更快？**

因为每个 Agent 的上下文只装一种「思维模式」。建筑师的脑子里是接口和数据流，施工队脑子里是变量和逻辑，质检员脑子里是 edge case 和断言。混在一起，Agent 会像没睡醒一样来回切换，效率暴跌。

---

## 技巧二：「异步任务」——让 Agent 在后台帮你查东西

另一个超实用的场景：你在主窗口写代码，突然需要查个东西。

**别在主窗口问！** 开一个**新会话**专门查：

> "这个 npm 包的最新版本有什么 breaking changes？"

等它查完、总结好，你直接复制结论，关掉那个会话。主窗口的上下文完全没被污染，思路没断。

**Claude Code 的 `/terminal` 或新终端里跑 `claude` 都能开新会话。**

**Kimi Code** 直接新开窗口就行，更简单。

---

## 技巧三：「备份会话」——改代码前先存档

要动一个核心模块？怕 Agent 改崩？

**操作：**
1. 先开一个**备份会话**，让 Agent 把当前文件状态总结一下，存个 `BACKUP.md`
2. 在主会话里大刀阔斧地改
3. 改崩了？回到备份会话，让它帮你恢复

这比 `git stash` 更灵活，因为 Agent 能**理解语义**——你说"恢复成那个带缓存的版本"，它知道你说的是哪个，不用你翻 commit hash。

---

## ⚠️ 注意：别「三开」变「三懵」

多开不是越多越好。建议**最多 3 个**，且每个有明确分工。否则你会：
- 忘了哪个窗口在干嘛
- 在 A 窗口问了 B 窗口的事
- 最后比单窗口还混乱

**一个口诀**：
> 🎯 三开有分工，单开有专注，乱开必翻车。

## 今日行动建议

今天写代码时，试试这个：

1. 开一个「建筑师」窗口，把你要做的功能接口先设计清楚
2. 开一个「施工队」窗口，按设计实现
3. 实现完，让「质检员」窗口 review

你会发现 Agent 的「智商」好像突然变高了——其实它只是**没被打扰**。
