## 先讲个恐怖故事

你入职第一天，leader 拍拍你肩膀：

> *「这个项目是 2019 年开始写的，核心模块是个叫 `legacy_core_v2_final_REALLY.js` 的文件，你先看懂它。」*

你打开文件，5000 行，没有注释，变量名是 `a1`、`b2`、`temp3`，函数名是 `doStuff()` 和 `processThing()`。

**你盯着屏幕，感觉不是在读代码，是在考古。** 🏺

别慌。Agent 在这方面是专业的——它不怕屎山，因为它没有嗅觉。

## 技巧一：让 Agent 画「代码地图」

人脑不适合直接消化几千行扁平代码，但擅长看图。让 Agent 帮你把代码结构翻译成可视化的地图。

**操作**：
> "请分析这个文件的代码结构，用 Mermaid 图画出：
> - 主要函数之间的调用关系
> - 数据流向
> - 哪个函数是入口，哪个是核心逻辑"

**Claude Code 版**（更省事）：
```
/mermaid
```
它会自动分析当前文件，生成调用关系图。

**你会得到什么**：
一张像地铁线路图一样的结构图，一眼就能看出：
- 哪块是「终点站」（没有下游调用的叶子函数）
- 哪块是「换乘站」（被多个函数调用的公共逻辑）
- 哪条线特别长（嵌套过深的调用链，大概率是重灾区）

💡 **进阶玩法**：让 Agent 把图存成 `ARCHITECTURE.md`，下次新会话直接说「先读 ARCHITECTURE.md」，秒进状态。

## 技巧二：让 Agent 写「代码考古报告」

不要直接问"这段代码干嘛的"——问太笼统，答案也笼统。

**用考古学家的思路提问**：

> "假设你是 2030 年的考古学家，正在研究这段 2019 年的代码。请写一份『考古报告』，包括：
> 1. 这段代码的『用途推测』（它想解决什么问题）
> 2. 『地层分析』（哪些是新加的补丁，哪些是原始逻辑）
> 3. 『未解之谜』（有哪些看不懂的魔法数字、奇怪判断、无注释的 hack）
> 4. 『危险区域』（改哪里最可能引发连锁反应）"

**为什么这种问法更管用**：
- 给 Agent 一个**角色设定**，它的回答会更结构化
- 「地层分析」逼它区分核心逻辑和后期补丁——后者往往是 bug 温床
- 「危险区域」相当于让 AI 先帮你踩一遍雷

**真实案例**：
我用这招分析过一个老项目的权限模块，Agent 揪出一个隐藏逻辑：

> *"注意第 347 行的 `if (user.id === 42)`——这看起来是调试时硬编码的管理员 ID，但没有任何注释说明。如果未来 42 号用户被删除或重建，这条判断可能失效。"*

**这玩意人工审查，十个人里有九个会跳过。** 但 Agent 对「奇怪的数字」特别敏感。

## 实战组合技 🥊

下次遇到祖传代码，按这个顺序来：

1. **先航拍**（让 Agent 画代码地图）—— 搞清楚整体结构，别一头扎进细节
2. **再勘探**（让 Agent 写考古报告）—— 标出危险区域和未解之谜
3. **最后动手**（只在 Agent 标出的「安全区」里改）—— 祖传代码的第一原则是：能跑就别动

---

## ⚠️ 祖传代码生存法则

Agent 能帮你读懂，但**别让它直接改**。原因有三：

1. **它没有恐惧**——你可能知道「这行代码看起来多余，但删了生产环境会炸」，AI 不知道
2. **测试覆盖率可能是 0**——改了没法验证，等于盲狙
3. **业务逻辑可能在注释里**——而注释，早就没了

**正确姿势**：让 Agent 帮你理解，改之前自己再确认一遍。
