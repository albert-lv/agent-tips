# 🎯 每日 Agent 技巧 · 2026-06-28

> 致：向上 & 小惠 —— 今天聊一个「把车交给 AI 开」的正确姿势

---

## 技巧一：别让 Agent 一脚油门踩到底

很多人用 Agent 的方式是这样的：

> *「帮我重构整个用户认证模块。」*

然后 AI 吭哧吭哧改了 20 个文件，删了 800 行加了 1200 行，你看着 diff 陷入沉默——**这不是重构，这是重写。**

### 问题在哪？

Agent 是「执行型」选手，不是「规划型」选手。你给的目标越大，它越倾向于「既然你信我，那我就放开手脚干」——然后你就收获了一个和你预期差距十万八千里的结果。

### 解法：渐进式委托（Gradual Handoff）

把一个大的任务拆成 **4 个阶段**，每个阶段都让 Agent 停下来汇报：

| 阶段 | 你做什么 | Agent 做什么 | 检查点 |
|------|---------|-------------|--------|
| **1. 需求对齐** | 描述问题/目标 | 复述理解、列出影响范围 | ✅ 确认影响范围 |
| **2. 方案设计** | 给约束条件 | 输出改动计划（不改代码） | ✅ 确认方案可行 |
| **3. 分批实现** | 指定第一批改哪些 | 改代码 + 跑测试 | ✅ 测试通过 |
| **4. 迭代推进** | 确认继续 | 继续下一批 | ✅ 全部完成 |

### 实战话术

❌ **错误示范**：
> 「优化这个接口的性能。」

✅ **正确示范**：
> 「这个接口响应慢，先别改代码。第一步：帮我分析可能的瓶颈在哪（SQL？序列化？还是网络？）。分析完汇报，我确认方向后再动手。」

等 Agent 分析完：
> 「方向 OK。先只改数据库查询部分，其他不动。改完跑测试，通过再汇报。」

**为什么要这么麻烦？**
- Agent 不会「过度发挥」——你锁死了边界
- 每步都有测试兜底，出问题立即发现
- 你可以随时喊停或调整方向

💡 **Pro tip**：Claude Code 用户可以在关键步骤手动创建 git checkpoint；Cursor 用户可以用 Composer 的 Plan/Act 模式，先锁死在 Plan 模式里讨论清楚。

## 技巧二：让 Agent 当自己的「代码审查员」

Agent 写代码快，但写完之后它自己也经常「不知道哪里会出问题」。你可以让它换个身份再检查一遍。

### 具体操作

代码写完后，不要直接说「好了」，而是追加一句：

> 「现在请你扮演一个严格的代码审查员，用 Code Review 的标准检查刚才的改动。特别关注：
> 1. 有没有破坏原有 API 契约？
> 2. 错误处理是否完整？
> 3. 边界情况（空值、越界、并发）是否考虑到了？
> 4. 如果这是生产代码，你敢不敢直接合并？」

**为什么这招灵？**

Agent 在「写代码模式」和「审查代码模式」下的关注点完全不同。写的时候它 focus 在「实现功能」，审查的时候它会切换到「找问题」——同一个 AI，换个 prompt 就能帮你把一道关。

### 升级版：「红队测试」

如果改的是核心逻辑，可以让 Agent 再进一步：

> 「假设你是一个恶意用户，想尽一切办法让这个新逻辑出错。你会怎么构造输入？列出 3 个最可能翻车的情况。」

Agent 往往会说出一些你俩刚才都没想到的 edge case。

---

## 一句话总结

> **Agent 不是越「放手」越高效，而是越「分段」越靠谱。把大任务切成小步、每步设检查点、让 AI 自己 review 自己——这才是「自动驾驶」的正确打开方式。**
