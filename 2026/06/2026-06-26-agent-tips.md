# 🎯 每日 Agent 技巧 · 2026-06-26

> 致：向上 & 小惠 —— 今天聊一个每天都在发生但你可能没意识到的事

---

## 技巧一：让 Agent 先「报幕」再动手

你有没有过这种经历：

> 你跟 Agent 说「帮我优化这个函数」，然后它噼里啪啦改了一堆，你盯着 diff 看了半天，发现它把核心逻辑全换了，而你只是想改个变量名。

**问题出在哪？** 你给了 Agent **目标**，但没给 **约束**。

### 解法：「报幕式」指令

在让 Agent 动手前，先让它口头报告它的计划：

> *「先别改代码，告诉我你打算怎么改，改哪几行，预期效果是什么。」*

等 Agent 报完幕，你确认方案 OK 了，再说：

> *「按刚才说的执行。」*

**为什么这招好用：**
- Agent 不会擅自「发挥创意」——它已经有了计划，后续执行会更克制
- 你有机会在代码被改之前 veto（否决）不靠谱的方案
- 相当于给 Agent 加了一个「确认对话框」

💡 **Pro tip**：Claude Code 用户可以用 `/preview` 或先描述计划再执行；Cursor 用户在 Composer 模式下可以先让 AI 生成「Plan」再执行「Act」。

## 技巧二：Diff 不是「看全篇」，是「看边界」

Agent 改完代码后，你面对一个巨大的 diff 页面，感觉像在审查毕业论文。

**别这样看。按这个顺序来：**

### 第一眼看「边界」
- 哪些**函数签名**被改了？（影响调用方）
- 哪些**导入语句**被增删了？（可能有依赖变更）
- 有没有**全局变量**或**配置文件**被碰了？（影响范围最大）

### 第二眼看「控制流」
- `if` / `else` / `for` / `while` 的结构有没有变？
- `return` 的位置有没有挪？（很容易出 bug）
- 异常处理 `try/catch` 有没有被改？（Agent 有时会「优化」掉它认为多余的异常处理）

### 第三眼才是「具体实现」
- 变量名、算法细节、注释……这些最后再看

**为什么这个顺序安全？**
因为 Agent 在「具体实现」上通常很靠谱，但在「边界处理」上容易翻车——比如删掉一个它认为「没用」的参数，结果上游调用方炸了。

---

## 一句话总结

> **先让 Agent 报幕，再按「边界→控制流→实现」的顺序审 diff。用 Agent 不是放任它干，而是把「动手权」给它，把「决策权」留给自己。**
