# 🦆 Agent 的「橡皮鸭调试」—— 为什么对着 AI 解释代码，bug 就自己出来了

> 日期：2026-06-23（周二）

## 先讲个程序员老梗

传说中，每个程序员桌上都该有一只橡皮鸭。遇到 bug 卡住时，就对着鸭子一行行解释代码。说着说着，突然一拍大腿：

> *「啊！我懂了！是这里少了个 break！」*

这只鸭子全程没说话，但你**在解释的过程中，自己就把问题想通了。**

这叫 **Rubber Duck Debugging（橡皮鸭调试法）**——一个被写进维基百科的正经技巧。

**好消息：现在你可以把那只鸭子换成 Claude Code / Kimi Code / Cursor，效果比鸭子好 10 倍。** 🦆➡️🤖

## 技巧一：「说人话」—— 让 AI 逼你把逻辑翻译成自然语言

bug 之所以难找，往往是因为你**太熟悉**这段代码了，大脑自动跳过了关键步骤。

**操作**：复制一段你怀疑有问题的代码，跟 Agent 说：

> "请用大白话解释这段代码在干什么，每一步都讲清楚。"

**为什么有效**：
- 你被迫重新理解自己的代码，而不是靠"肌肉记忆"
- AI 的翻译会暴露你模糊的地方——"这里看起来是先过滤再排序，但其实…"
- 经常说到一半你自己就发现了：「等等，我好像搞反了顺序」

**Claude Code 版**（更直接）：
```
/explain
```
它会自动分析当前文件或选中的代码，用段落式人话解释逻辑。

## 技巧二：「扮演面试官」—— 让 AI 拷问你的假设

有时候你知道代码有问题，但不知道问题在哪。这时候换个角度：让 AI 当你的面试官。

**操作**：
> "你是一位正在面试我的 senior engineer。这段代码是我刚写的，请你逐行 review，找出所有潜在的 bug 和 edge case。"

**你会发现 AI 会揪出这些**：
- "如果输入为空数组，这里会报错吗？"
- "这个循环有没有可能越界？"
- "并发情况下这个变量安全吗？"

**这些问题其实就是你自己应该问自己的**——但让 AI 问出来，更容易触发你的"防御本能"，从而主动修复。

💡 **进阶玩法**：让 AI 生成一份「面试 checklist」，下次写代码前先看一遍，写完再对照检查。

## 技巧三：「假设反转」—— 让 AI 帮你质疑一切

最隐蔽的 bug 往往是那些"不可能出问题"的地方。

**操作**：
> "假设这段代码有 bug，但我不知道在哪。请你列出**最不可能出错的 3 个地方**，并说明为什么它们其实可能有问题。"

**反向提问的力量**：
- 人的直觉是"这里肯定没问题"，但 bug 往往藏在这种自信里
- AI 没有先入为主，它会一视同仁地检查所有路径
- 经常你会发现：「啊，我从来没想过这个函数在失败时会返回什么」

## 实战组合技 🥊

下次遇到 debug 死局，按这个顺序来：

1. **先解释**："AI，请用大白话解释这段代码"
2. **再面试**："扮演 senior，review 这段代码"
3. **最后反转**："假设这里有 bug，最不可能在哪？"

三步走完，90% 的 bug 会自己现身。剩下的 10%，至少你已经排除了大量干扰项。

---

## 一句话总结

> **Agent 不是替你 debug，而是逼你用「说人话」的方式重新理解代码——而这个过程，就是 debug 本身。**

今天试试这个：下次遇到 bug 不要急着加 console.log，先找个 Agent 窗口说「请解释这段代码」。

你会发现，有时候 bug 不是被找到的，而是**被说出来的** 🦆
